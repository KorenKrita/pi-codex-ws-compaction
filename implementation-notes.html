Implementation Notes

This file records implementation-relevant decisions that were not fully determined before live integration.

Effective authorization headers override the resolved API key

Provenance: agent-resolved

Pi can provide an effective Authorization header alongside an apiKey value. Header names are case-insensitive, but plain JavaScript objects are not. The first live compaction test therefore sent both authorization and Authorization; the gateway selected the placeholder credential and returned HTTP 401.

The compaction client now merges headers case-insensitively, allowing Pi's later effective provider header to replace the generated bearer header. An offline regression assertion verifies there is exactly one authorization header, and the live compaction suite verifies the corrected request.

No silent HTTP continuation for normal target-model turns

Provenance: agent-resolved

Normal configured-model turns fail visibly when WebSocket connection or delivery fails. This avoids silently switching to HTTP previous_response_id, whose continuity was disproved on the target gateway. Pi-internal summarization calls are the sole exception: they have no session id and use full-input HTTP without previous_response_id.

Configuration is snapshotted at extension load

Provenance: agent-resolved

The provider override, model allowlist, and WebSocket endpoint are treated as one runtime configuration. Reading the file independently inside request hooks could mix old and new values during an edit, while changing the provider cannot safely re-register the extension mid-request. The extension therefore validates and captures one configuration at load time; changes take effect through Pi /reload. Setting enabled: false before reload causes the extension to register nothing.

Continuation offsets use resolved context-message counts

Provenance: agent-resolved

Pi session entries and model-visible context messages are not guaranteed to have a one-to-one relationship. WebSocket incremental continuation therefore records offsets using sessionEntryToContextMessages over the compaction-aware context entries rather than counting raw branch message entries. This prevents a future context-expanding entry from shifting the slice and duplicating or dropping input.

Remote artifacts seed WebSocket continuity once

Provenance: agent-resolved

After compaction, the first compatible request replays the opaque v2 artifact and retained history. Once that request completes, later requests on the same socket use its previous_response_id and send only new context messages. Replaying the full artifact history on every turn would defeat WebSocket continuity and allow explicit history to grow without bound.

Legacy v1 artifacts are intentionally not parsed or replayed because the target gateway does not support the legacy endpoint and this repository is scoped to v2 only.
